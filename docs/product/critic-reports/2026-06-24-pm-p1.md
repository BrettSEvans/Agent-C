### 6. Omissions — ⚠️ MINOR ISSUES (severity: minor)

Three gaps that downstream stages will need to address:

**6a. Platform requirements ambiguity**

The brief states Agent-C "runs it locally in their own Claude setup" and is "built on the Claude Code/Desktop skills & subagents ecosystem." This conflates two different platforms:

- **Claude Desktop:** has skills, but NOT subagents.
- **Claude Code:** has both skills and subagents.

The orchestration model (dispatching subagents to stage agents, managing stage registry, presenting approval gates) requires **subagents**, which only Code provides. Desktop can invoke skills directly but cannot dispatch subagents.

**Clarification needed (for architect):** Is v1 Code-only? Or is there a Desktop-only fallback path (with manual orchestration)? Currently ambiguous.

**6b. Model selection / Claude version support**

The brief doesn't specify what Claude models are supported or required. Does Agent-C target Claude 3.5 Sonnet+? Will it work with older models or non-Anthropic backends? This affects:
- Installation/distribution messaging
- User expectations around capability
- Whether "bring your own Claude subscription" implies Sonnet+

**Clarification needed (for architect):** Lock in minimum Claude model version.

**6c. Mid-stage resumption interaction**

The brief promises users can "resume any one at any lifecycle stage," but doesn't clarify the resumption UX:
- If an agent's elicitation is interrupted mid-flow, can the user pick it up again at the same stage?
- Does the system save conversation state, or just the final artifact?
- How is "I already answered this" handled if the user re-runs a stage?

**Clarification needed (for UX/architect):** Design the resumption interaction and save/restore model.

### 7. Errors — ✅ PASSES

### 8. Oversights — ⚠️ MINOR ISSUES (severity: minor)

Two gaps that reflect edge-case thinking:

**8a. Skill extension / customization model**

The brief emphasizes "skills the developer owns, editable and version-controlled — a toolkit, not a black box." But it doesn't describe HOW users extend or customize skills:

- Git fork the repo and edit inline?
- Hooks or plugin interface?
- Load custom skills from a separate folder?
- Override behaviors via config?

This is important for the "toolkit, not black box" promise, but the mechanism is undefined.

**Clarification needed (for design/architect):** Define the extensibility model so users understand how to customize Agent-C.

**8b. Failure modes under registry/artifact tampering**

The brief promises "context never lost" and "versioned docs flow agent-to-agent," but doesn't address edge cases:

- What happens if a user manually deletes a doc or corrupts the registry file?
- If the registry gets out of sync with actual project folders, what's the recovery?
- Can a user revert a stage to a prior version if they changed their mind?

These are edge-case failure modes, not core-product misses, but worth documenting for QA/architect.

**Note (for architect/QA):** Document failure modes and recovery paths (or accept as limitations of v1).
