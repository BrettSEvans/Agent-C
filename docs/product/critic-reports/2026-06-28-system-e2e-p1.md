## Executive summary

Agent-C is internally coherent at the **skill** layer: the 13 skills share a clean
skeleton, the lifecycle and artifact chain are well specified, and the contracts
(recommend-don't-chain, dual-mode, backlog-as-artifact) are consistently stated. The
problems are at the **system-documentation and dogfooding** layer:

1. **Agent-C has not run its own full process on itself.** The product whose entire
   value proposition is "every stage produces a reviewable artifact you approve before
   the next begins" stops its *own* discovery chain at stage 2. There is no `03`, `04`,
   or `05` for the Agent-C system. (Significant)
2. **The platform/GUI story has drifted out from under the brief.** The PM brief locks
   v1 as "Claude Desktop, no subagents, text/conversational only, **no GUI**." The repo
   now contains a full **Electron GUI dashboard**, **Code subagent wrappers**
   (`agent-defs/`), and a README that says "Desktop **or** Code." The system artifacts
   were never updated to reflect this evolution. (Significant)
3. **The dashboard is invisible at the system level.** The single largest piece of
   built software in the repo (`dashboard/`) is not mentioned anywhere in
   `ARCHITECTURE.md` — not in the skill catalog, the lifecycle, or the §10 repo layout.
   (Significant)
4. **A new cross-surface requirement (your point 1) has no home in the artifacts.**
   "An action surfaced on one surface must surface on every surface" is a real product
   rule, partially implemented in the orchestrator already, but not captured in the
   UX workflow or anywhere a future maintainer would find it. (Significant)

The seven minor findings are drift/staleness items: build-status claims that no longer
match the code, a dated model-version claim, and an internal inconsistency about the
critic model after the recent auto-apply change.

## Findings by criterion

### 1. Discipline best practices — ✅ PASSES (skill layer)

The skills follow their own craft standards well. Every role skill carries the §4
skeleton (frontmatter → analysis → modes → product-type → themes → output → handoff →
customizing). The shared methods (`elicitation`, `best-practices`, `stage-protocol`,
`feature-mode`) are genuinely reused, not copy-pasted. The brief grounds itself in real
jobs-to-be-done ("idea → shipped without holding the whole process in your head"). No
issue here.

### 2. Brief consistency — ⚠️ SIGNIFICANT

**2a. The brief's platform & GUI constraints are now contradicted by the built system.**
`docs/product/01-pm-brief.md:6` — *"designed for **Claude Desktop** (not Code) …
no GUI screens."* `02-ux-workflow.md:6` — *"Agent-C … has **no GUI screens**"* and
`:75` — *"Text/conversational only … no GUI."* Yet:
- `dashboard/` is a full Electron + React GUI app.
- `agent-defs/` exists specifically to enable **Claude Code subagent** dispatch.
- `README.md:66` — *"invoke a stage by name in **Claude Desktop or Claude Code**."*
- `README.md:71-73` explicitly describes the Code/subagent path.

This is healthy product evolution, but the **brief and UX workflow were never updated**,
so the source-of-truth documents now actively misstate what the product is. A reader
starting from `01-pm-brief.md` would conclude there is no GUI and no Code support.

*Fix:* Add a "Post-v1 evolution" or "Platform update" section to `01-pm-brief.md` and
`02-ux-workflow.md` recording that (a) the product now spans Desktop (manual) + Code
(subagent wrappers) and (b) the text dashboard has a GUI counterpart (the Electron
dashboard), with the original text-only framing marked as the v1-discovery baseline.

### 3. Prior-artifact coherence — ⚠️ SIGNIFICANT

**3a. The system's own artifact chain is broken after stage 2.** `02-ux-workflow.md:121`
hands off: *"UI agent → … writes docs/product/03-ui-direction.md."* That file does not
exist. Neither does `04-architecture.md` nor `05-implementation.md` at the system level.
`ARCHITECTURE.md:233-234` quietly concedes this: *"Agent-C has been run on itself:
docs/product/ holds 01-pm-brief.md, 02-ux-workflow.md, backlog.md, wireframes/, and
critic-reports/."* — i.e., 2 of 6 stages.

This is the central dogfooding gap. It is defensible (the system was built by hand, not
by running itself end-to-end), but it should be **stated explicitly as a known
limitation**, not left as a dangling "Next handoff" that implies the chain completed.

*Fix (choose one):* either (a) generate the missing `03`/`04`/`05` for the Agent-C
system by running the UI/architect/engineer skills on it, or (b) add an explicit
"Dogfooding status" note to `02-ux-workflow.md` and `ARCHITECTURE.md` recording that the
system's own chain intentionally stops at UX and why, so the broken handoff isn't read
as an accident.

### 4. Completeness — ⚠️ SIGNIFICANT

**4a. The dashboard sub-product is absent from `ARCHITECTURE.md`.** `ARCHITECTURE.md`
bills itself as *"the top-level map"* (`:8`) and the §10 repo-layout tree (`:260-291`)
omits `dashboard/` entirely. The dashboard has a complete, high-quality discovery chain
of its own (`dashboard/docs/product/01`–`05` + critic-reports + qa-report), yet a reader
of the system map would never know it exists or how it relates to the orchestrator.

*Fix:* Add a short "Sub-products" or "The dashboard" section to `ARCHITECTURE.md` and add
`dashboard/` to the §10 tree, framing it as the GUI realization of the orchestrator's
text dashboard touchpoint (it already self-describes this way in
`dashboard/docs/product/01-pm-brief.md:10`).

**4b. README "Status" omits two shipped capabilities.** `README.md:75-87` describes v1 as
complete but does not mention (a) the Electron dashboard at all, or (b) the
technical-critic **auto-apply** model just introduced. A new user reading Status gets an
incomplete inventory.

### 5. Quality & clarity — ✅ PASSES

The prose is consistently clear and specific. Decisions are marked confirmed vs. assumed
(`01-pm-brief.md:102-143`). The "only this product" and "best practice is craft, not
sameness" framings in `ARCHITECTURE.md:160-184` are distinctive and well argued. No
vagueness issue.

### 6. Omissions — ⚠️ SIGNIFICANT + minor

**6a. (SIGNIFICANT) The cross-surface action-notification rule is undocumented.** Your
point 1 — *"whenever an action for the user is returned, it must be communicated on
every surface (dashboard, Claude Desktop, and a future Claude Code CLI) so the user
knows there's an action no matter what surface they're on"* — is a genuine product
requirement. It is **partially implemented** (the orchestrator now emits a "What needs
your attention" block, and the dashboard computes a Claude prompt), but:
- It is not stated in `02-ux-workflow.md` §4 (States & feedback), where the "needs-you
  flag" lives (`:56-58`) — the natural home for it.
- The symmetry is incomplete: chat→user is covered, but the principle that the
  **dashboard** must also always show the same pending action (and that the two must not
  diverge) is not written down anywhere as an invariant.

*Fix:* Add an explicit "Cross-surface notification" requirement to `02-ux-workflow.md`
§4 and a corresponding contract line in the `orchestrator` skill: *every `needsYou:true`
entry must be surfaced on every active surface; surfaces read the same `state.json` so
they cannot diverge.*

**6b. (minor) No documented recovery for registry/state corruption at the system level.**
The original PM critic report (`critic-reports/2026-06-24-pm-p1.md:107-117`) already
flagged "failure modes under registry/artifact tampering" and it was noted for
architect/QA — but with no system `04`, it was never designed. Still open; re-flag.

### 7. Errors — ⚠️ minor

**7a. Internal inconsistency on the critic model after the auto-apply change.**
`README.md:82-83` (Status) says only *"Critic is integrated as an opt-in gate action on
PM/UX/UI stages"* and says nothing about technical-critic. `ARCHITECTURE.md:218-221` now
says technical-critic findings are **auto-applied with no HITL**. These two top-level docs

**7b. (minor) Dated model claim.** `01-pm-brief.md:8` and `:119` fix the model floor at

Opus 4.8). Not wrong as a historical floor, but if the brief is meant to be current
guidance it reads as stale. Flag lightly (knowledge-cutoff-sensitive).

### 8. Oversights — ⚠️ minor

**8a. "Recommend, don't auto-chain" vs. the new auto-apply critic loop.**
`ARCHITECTURE.md:36-38` states the locked principle *"No stage invokes the next
directly."* The new technical-critic→engineer auto-apply path (`technical-critic`
writes `pendingFeedback`, engineer applies it without a gate) is *adjacent* to that
principle. It does not violate it (the critic isn't a lifecycle stage and the engineer
is reacting to feedback, not being auto-invoked), but the tension is unaddressed in the
principles section. A one-line carve-out would prevent a future reader seeing a
contradiction. (minor)

**8b. `agent-defs/` README status vs. reality.** `agent-defs/` is described as *"idle
until autonomous dispatch is built (post-v1)"* (`ARCHITECTURE.md:227`). Worth a quick
confirm that the stubs there still match the current skill contracts after the recent
engineer/technical-critic edits (the technical-critic and engineer stubs *were* updated
in `bca7701`; the others were not re-checked this pass).
