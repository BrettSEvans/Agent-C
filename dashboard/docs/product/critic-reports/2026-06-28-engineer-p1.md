## Summary

The implementation is solid overall: the Electron/React/TypeScript/Zustand stack is faithfully built, the typed preload bridge is clean, the DI pattern on git-service makes the test suite meaningful, and TDD coverage is real (not theater). The security model (contextIsolation ON, nodeIntegration OFF) is correctly applied.

Two significant issues need fixing before QA:

1. A resource leak: the watcher is recreated without closing the old one every time `list-projects` is invoked.
2. A security regression: `sandbox: false` is set unnecessarily, weakening the renderer sandbox without benefit.

Four minor issues are also noted.

## Issues Found

### SIGNIFICANT — Criterion 4/7: Watcher resource leak

**File:** `src/main/services/watcher-service.ts:13` and `src/main/ipc/handlers.ts:29`

`WatcherService.start()` overwrites `this.watcher` with a new chokidar instance without calling `close()` on the previous one:

```ts
// watcher-service.ts — start() as written:
this.watcher = chokidar.watch(pathsToWatch, { ... })
// old watcher is abandoned without close()
```

`list-projects` calls `startWatching(entries)` on every invocation. The renderer calls `list-projects` on mount, on every `registry-changed` event (App.tsx:88), and on Refresh All. Each call accumulates a new chokidar watcher. The old watchers keep firing `change` events indefinitely, multiplying event emissions and leaking file descriptors.

**Fix:** Guard `start()` against double-start — either close the existing watcher first, or return early if already running:

```ts
start(registryPath: string, statePaths: Map<string, string>): void {
  this.watcher?.close()        // close existing before creating a new one
  this.statePaths = statePaths
  // ... rest unchanged
}
```

Alternatively, call `startWatching` only once (on first `list-projects`) and use `addStateWatch`/`removeStateWatch` for incremental updates thereafter.

### SIGNIFICANT — Criterion 1/4: `sandbox: false` weakens renderer security unnecessarily

**File:** `src/main/index.ts:15`

```ts
webPreferences: {
  preload: join(__dirname, '../preload/index.js'),
  contextIsolation: true,
  nodeIntegration: false,
  sandbox: false          // ← unnecessary, degrades security
}
```

The architecture states the security model as "context isolation on, node integration off." It says nothing about disabling the sandbox. The preload (`src/preload/index.ts`) uses only `contextBridge` and `ipcRenderer` — both are available in sandboxed mode. There is no reason to disable the sandbox.

With Electron 20+, the sandbox is enabled by default precisely because it limits what a compromised renderer can do to the host OS. Setting `sandbox: false` removes that OS-level isolation. This app reads arbitrary filesystem paths from user-controlled registry data, making renderer isolation meaningful.

**Fix:** Remove `sandbox: false` (let it default to `true`). If the preload fails to load under sandbox, add `sandbox: true` explicitly and verify the preload works — which it should, since it only uses Electron context APIs.

### Minor — Criterion 4: Missing `uncommittedFiles` in two failed/computing GitState literals

**Files:** `src/renderer/src/App.tsx:33` and `src/renderer/src/App.tsx:127`

Two places construct a `GitState` object without the `uncommittedFiles` field, which is required per the `GitState` type in `shared/types.ts`:

```ts
// App.tsx:33 — on fetch error
updateGitState(entry.path, {
  uncommittedCount: 0,
  unpushedCount: 0,
  computedAt: new Date().toISOString(),
  status: 'failed'              // missing: uncommittedFiles: []
})

// App.tsx:127 — on refresh start
updateGitState(path, {
  uncommittedCount: 0,
  unpushedCount: 0,
  computedAt: new Date().toISOString(),
  status: 'computing'           // missing: uncommittedFiles: []
})
```

`ProjectDetail` guards against this with `gitState.uncommittedFiles ?? []`, which is why it doesn't crash, but the type contract is violated. If `tsc --noEmit` truly exits 0, the tsconfig may not be strict enough to catch missing required properties.

**Fix:** Add `uncommittedFiles: []` to both objects. Optionally audit the tsconfig for `strict: true`.

### Minor — Criterion 7: `lastKnown` fallback described in architecture but not implemented

**File:** `src/main/services/git-service.ts:68-76`

`04-architecture.md §8` describes the error fallback as: "try fresh → on failure fall back to `lastKnown` git state with a warning + age." The `GitState` type has an optional `lastKnown` field for this purpose. In the implementation, the git-service returns `{ status: 'failed', ...zeros }` on error — `lastKnown` is never populated.

The UI shows "Git status unavailable" on failure with no prior-state context, which is a minor degradation vs. the spec. Not a crash risk, but the stale fallback display was called out as a user-facing feature in the UX and architecture.

**Fix (if desired for v1):** In `get-git-state` IPC handler, before calling `gitService.getGitState`, read the existing cached value and populate `lastKnown` in the failure return. Or pass the cached value into `createGitService` as a fallback provider.

This is low priority — mark as a v1 backlog item if not fixing now.

### Minor — Criterion 5: `removeStateWatch` is never called; stale watchers accumulate for deleted projects

**File:** `src/main/ipc/handlers.ts:90-95`

The `registry-changed` handler adds watchers for new projects but never removes watchers for projects that were removed from the registry:

```ts
watcher.on('registry-changed', async () => {
  win.webContents.send('registry-changed')
  const entries = await loadAllProjects()
  for (const entry of entries) {
    watcher.addStateWatch(entry.path, getStatePath(entry.path))   // adds
    // never calls watcher.removeStateWatch for removed entries
  }
})
```

`WatcherService.removeStateWatch()` exists but is dead code. If a project is removed from the registry, its `state.json` watcher keeps firing. Low severity (the event just fails to match in the renderer), but it's a leak.

**Fix:** Compute the set difference between the current `statePaths` keys and the new entries, then call `removeStateWatch` for each removed path.

### Minor — Criterion 6: `window.api` type locally redeclared in App.tsx instead of importing `DashboardAPI`

**File:** `src/renderer/src/App.tsx:9-23`

App.tsx declares a local `window.api` interface that duplicates the `DashboardAPI` shape from `src/preload/index.ts`. If the preload API surface changes, the local declaration won't update automatically — TypeScript won't catch the drift.

**Fix:** Export `DashboardAPI` from `src/preload/index.ts` and import it in App.tsx:

```ts
// App.tsx
import type { DashboardAPI } from '../../preload'

declare global {
  interface Window {
    api: DashboardAPI
  }
}
```

(Requires the renderer tsconfig to include the preload path, or move `DashboardAPI` to `shared/types.ts`.)
